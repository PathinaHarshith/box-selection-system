# AI Usage Disclosure

This document discloses the usage of AI assistance in the design, implementation, and verification of the Django Box Selection System.

---

## 1. Tools Used
- **AI Tool**: Claude (Anthropic), via the claude.ai chat interface / Claude in a coding environment.
- **Model Family**: Claude 3.5 Sonnet.

---

## 2. Prompts Given
A detailed specification prompt was provided to the AI. The prompt detailed all project requirements, including:
- Database models (`Box`, `Product`, `Order`, `OrderItem`) and list view registrations in Django Admin.
- The `PackingService` core logic integrating `py3dbp` with a cost-first selection criteria and volume-utilization tie-breaker.
- Serializers validation rules (handling quantities and unregistered SKUs) and views utilizing correct serialization objects.
- Integration tests covering volumetric fits, weight violations, cost minimizations, unpackable items, and malformed payloads.
- CI pipelines via GitHub Actions and static document templates.

---

## 3. Accepted Output
The following outputs generated by Claude were accepted:
- The complete Django project structure (`box_selection/`) and directory layout.
- The model definitions using `DecimalField` to prevent floating-point drift.
- The `PackingService` wrapper using `py3dbp` for 3D coordinate space calculations.
- Serializers, views, and urls configured for OpenAPI/Swagger generation.
- The unit and integration tests inside `packing/tests/` verifying all 6 core testing scenarios.
- The GitHub Actions workflow file, seed fixtures, and detailed documentation.

---

## 4. Modified/Rejected Output
No code outputs were rejected; the initial codebase generation was accepted directly and subjected to verification and adjustment.

---

## 5. Mistakes Found and Corrected
1. **Insecure DEBUG Default**: The generated settings file originally defaulted `DEBUG` to `True` if the environment variable was missing. This was insecure and was corrected to default to `False`.
2. **Simplified Unpackable Reason Heuristic**: The check to determine the exact reason for an item being unpackable is a simplified heuristic checking if items individually fit any box's dimensions or weight capacities. It does not perform a full multi-item 3D packing simulation to identify the exact subset that causes a packing overflow.

---

## 6. Verification Steps
1. **Automated Tests**: Ran the local test suite using SQLite:
   ```bash
   python manage.py test --verbosity=2
   ```
   Verified that all 14 tests pass successfully.
2. **Manual Integration Testing**: Tested `/api/v1/pack/` manually using `curl` for successful packing scenarios, unpackable cases (such as the giant widget), and malformed inputs (invalid quantities and missing fields).
3. **API Documentation**: Checked the Swagger UI locally at `/api/docs/` to confirm that the endpoint descriptions, parameter lists, and schemas load correctly.
